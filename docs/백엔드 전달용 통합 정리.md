# 백엔드 전달용 통합 정리

## 전달 목적

이 문서는 LG Care Vision 프론트엔드에서 현재 구현된 채팅 흐름과 mock 데이터 구조를 백엔드 팀원이 한 번에 이해할 수 있도록 정리한 통합본입니다.

기존 문서가 여러 개로 나뉘어 있어 헷갈릴 수 있으므로, 백엔드 전달 시에는 이 파일 하나만 기준으로 보면 됩니다.

## 현재 프론트 상태

현재 프로젝트는 Figma Make 기반 React + Vite 프론트 코드입니다.

프론트에는 아직 실제 백엔드와 DB가 연결되어 있지 않고, 화면 안에 있던 임시 데이터를 아래 구조로 분리하는 작업을 진행했습니다.

```txt
src/app/
  api/
    client.ts
    chat.ts
    devices.ts
    user.ts
  data/
    mockChat.ts
    mockDevices.ts
    mockUser.ts
  types/
    chat.ts
    device.ts
    user.ts
```

현재 `api/*` 파일은 실제 서버를 호출하지 않고 mock 데이터를 반환합니다.
백엔드가 준비되면 프론트 화면을 크게 수정하지 않고 `api/*` 함수 내부만 실제 API 요청으로 교체하면 됩니다.

## 프론트와 백엔드 역할 구분

### 프론트엔드가 유지할 역할

- 화면 UI 렌더링
- 채팅 선택지 버튼 표시
- 현재 채팅 단계 관리
- 사용자가 선택한 값 임시 저장
- 제품군, 제품 종류, 증상 선택 흐름 처리
- LLM 호출 전 기본 시연 흐름 유지
- 선택 결과를 `chatContext`로 구성해서 백엔드 또는 LLM API에 전달
- 셀프케어 / AR / 영상 / 서비스센터 버튼 노출

### 백엔드가 맡을 역할

- 사용자 정보 조회
- 등록 제품 목록 조회
- 제품군 목록 제공
- 제품 종류 목록 제공
- 제품군/제품종류별 증상 목록 제공
- 증상별 추천 해결 방식 제공
- 채팅 메시지 저장
- 셀프케어 완료 기록 저장
- 서비스센터 신청 저장
- 추후 LLM 호출 처리

## 채팅 흐름

현재 채팅은 LLM이 모든 것을 판단하는 구조가 아니라, 프론트에서 선택지 기반으로 흐름을 먼저 좁힌 뒤 필요한 context를 백엔드 또는 LLM에 넘기는 구조입니다.

```txt
채팅 진입
→ 문의 목적 선택
→ 제품군 선택
→ 제품 종류 선택
→ 증상 선택
→ 셀프케어 / AR / 영상 / 서비스센터 / LLM 답변 분기
```

문의 목적:

```txt
제품 문제 해결
가전 관리 방법
서비스 센터 연결
```

제품군:

```txt
에어컨
냉장고
세탁기
공기청정기
```

제품 종류 예시:

```txt
에어컨: 스탠드형, 벽걸이형, 창문형
냉장고: 일반형, 양문형, 김치냉장고
세탁기: 드럼형, 통돌이형
공기청정기: 스탠드형, 벽걸이형
```

증상 예시:

```txt
전원이 불안정하거나 자주 꺼져요
냉방/기능이 잘 작동하지 않아요
필터에 먼지가 많이 쌓여 있어요
소음이나 진동이 심해요
그 외 다른 문제
```

## LLM에 넘길 채팅 context

프론트는 사용자가 선택한 값을 `chatContext`에 저장합니다.
이 값은 나중에 백엔드 API 또는 LLM API를 호출할 때 함께 넘기면 됩니다.

현재 프론트에 반영된 구조:

```ts
interface ChatContext {
  intent?: "trouble" | "care" | "service";
  productCategory?: string;
  productType?: string;
  deviceId?: string;
  productName?: string;
  model?: string;
  symptom?: string;
  issue?: string;
  recommendedActions?: ("manual" | "ar" | "video" | "service" | "llm")[];
}
```

예시:

```json
{
  "intent": "trouble",
  "productCategory": "냉장고",
  "productType": "양문형",
  "symptom": "소음이나 진동이 심해요",
  "recommendedActions": ["video", "manual", "ar", "service"]
}
```

정리하면, 제품군과 제품 종류를 고르는 것은 프론트의 UI 흐름입니다.
다만 제품군/제품 종류/증상 목록 데이터 자체는 백엔드에서 내려줄 수 있어야 합니다.

## 현재 프론트 API 파일

```txt
src/app/api/client.ts
src/app/api/chat.ts
src/app/api/devices.ts
src/app/api/user.ts
```

`client.ts`에는 추후 실제 서버 연결을 위한 기본 구조가 있습니다.

```ts
export const API_BASE_URL =
  import.meta.env.VITE_API_URL ?? "http://localhost:8080/api";
```

백엔드 연결 후에는 `.env` 또는 `.env.example`에 아래 값을 맞추면 됩니다.

```env
VITE_API_URL=http://localhost:8080/api
```

## 백엔드 필요 API

### 1. 사용자 정보 조회

```http
GET /api/users/me
```

응답 예시:

```json
{
  "id": "user-1",
  "name": "사용자",
  "phone": "010-0000-0000",
  "address": "서울특별시",
  "language": "ko"
}
```

### 2. 등록 제품 목록 조회

```http
GET /api/devices
```

응답 예시:

```json
[
  {
    "id": "1",
    "name": "거실 에어컨",
    "category": "에어컨",
    "model": "LG 휘센 벽걸이",
    "status": "정상"
  }
]
```

### 3. 제품군 목록 조회

```http
GET /api/product-categories
```

응답 예시:

```json
[
  { "id": "air-conditioner", "name": "에어컨" },
  { "id": "refrigerator", "name": "냉장고" },
  { "id": "washer", "name": "세탁기" },
  { "id": "air-purifier", "name": "공기청정기" }
]
```

### 4. 제품 종류 조회

```http
GET /api/product-categories/{categoryId}/types
```

응답 예시:

```json
[
  { "id": "standard", "name": "일반형" },
  { "id": "double-door", "name": "양문형" },
  { "id": "kimchi", "name": "김치냉장고" }
]
```

### 5. 증상 목록 조회

```http
GET /api/symptoms?categoryId=refrigerator&typeId=double-door
```

응답 예시:

```json
[
  { "id": "power-unstable", "label": "전원이 불안정하거나 자주 꺼져요" },
  { "id": "function-issue", "label": "냉방/기능이 잘 작동하지 않아요" },
  { "id": "filter-dust", "label": "필터에 먼지가 많이 쌓여 있어요" },
  { "id": "noise-vibration", "label": "소음이나 진동이 심해요" },
  { "id": "other", "label": "그 외 다른 문제" }
]
```

### 6. 증상별 해결 방식 조회

```http
GET /api/resolutions?symptomId=noise-vibration
```

응답 예시:

```json
{
  "symptomId": "noise-vibration",
  "summary": "소음 원인을 확인한 뒤 영상 가이드 또는 서비스 접수를 안내할 수 있습니다.",
  "recommendedActions": ["video", "manual", "ar", "service"],
  "videoId": "refrigerator-noise-guide",
  "requiresService": false
}
```

### 7. 채팅 메시지 저장

```http
POST /api/chat-messages
```

요청 예시:

```json
{
  "sessionId": "chat-session-1",
  "sender": "user",
  "message": "소음이나 진동이 심해요",
  "context": {
    "intent": "trouble",
    "productCategory": "냉장고",
    "productType": "양문형",
    "symptom": "소음이나 진동이 심해요"
  }
}
```

### 8. 셀프케어 완료 기록 저장

```http
POST /api/care-histories
```

요청 예시:

```json
{
  "deviceId": "1",
  "type": "Self Care",
  "title": "셀프 케어 완료",
  "completedAt": "2026-06-12T11:30:00+09:00"
}
```

### 9. 서비스센터 신청 저장

```http
POST /api/service-requests
```

요청 예시:

```json
{
  "deviceId": "1",
  "product": "에어컨",
  "model": "거실 에어컨 - LG 휘센 벽걸이",
  "issue": "냉방이 잘 되지 않아요",
  "customerName": "사용자",
  "phone": "010-0000-0000",
  "address": "서울특별시",
  "visitDate": "2026. 6. 13.",
  "visitTime": "09:00~11:00"
}
```

### 10. LLM 응답 생성

```http
POST /api/ai/chat
```

요청 예시:

```json
{
  "message": "소음이 계속 나요",
  "context": {
    "intent": "trouble",
    "productCategory": "냉장고",
    "productType": "양문형",
    "deviceId": "1",
    "productName": "주방 냉장고",
    "model": "LG 디오스",
    "symptom": "소음이나 진동이 심해요",
    "recommendedActions": ["video", "manual", "ar", "service"]
  }
}
```

응답 예시는 백엔드 구현 방식에 맞춰 정하면 됩니다.

```json
{
  "message": "냉장고가 벽이나 가구에 닿아 있으면 진동음이 커질 수 있습니다. 먼저 주변 간격을 확인해 주세요.",
  "recommendedActions": ["video", "service"],
  "requiresService": false
}
```

필수에 가까운 데이터:

- 기기 위험도 점수
- 관리 영상
- AR 가이드 매핑 정보
- LLM 대화 세션

## 프론트에서 이미 정리한 부분

완료된 작업:

- `Chat.tsx`의 주요 타입 분리
- 사용자 mock 데이터 분리
- 등록 제품 mock 데이터 분리
- 채팅 mock 데이터 분리
- API wrapper 파일 생성
- `chatContext` 추가
- 제품군, 제품 종류, 증상, 추천 액션을 context에 저장
- 채팅 localStorage 초기화 키 변경
- 빌드 성공 확인

현재 프론트는 mock 데이터를 쓰지만, API 연결을 위한 파일 구조는 만들어져 있습니다.

## 프론트에서 아직 남은 작업

다음 페이지들은 아직 일부 데이터가 화면 파일 안에 남아 있을 수 있습니다.

```txt
Device.tsx
DeviceDetail.tsx
SelfCare.tsx
VideoPlayer.tsx
Home.tsx 일부
```

주의할 점:

- Home 화면의 날씨 대시보드는 아직 수정 예정이라 이번 분리 작업에서 제외합니다.
- Home의 기기 위험도 데이터는 나중에 백엔드 연동 대상으로 볼 수 있습니다.
- 채팅 흐름 전체를 백엔드로 넘기는 구조는 추천하지 않습니다.
- 백엔드는 선택지와 저장 API를 제공하고, 프론트가 UI 흐름을 관리하는 구조가 현재 프로젝트에 맞습니다.

## 백엔드 팀에 전달할 핵심 요약

프론트 채팅 흐름은 선택지 기반으로 유지합니다.

백엔드는 아래 API를 우선 준비하면 됩니다.

```txt
GET  /api/users/me
GET  /api/devices
GET  /api/product-categories
GET  /api/product-categories/{categoryId}/types
GET  /api/symptoms
GET  /api/resolutions
POST /api/chat-messages
POST /api/care-histories
POST /api/service-requests
POST /api/ai/chat
```

프론트는 사용자가 선택한 제품군, 제품 종류, 증상 정보를 `chatContext`로 묶어서 전달합니다.

LLM은 처음부터 전체 채팅 흐름을 대신하는 것이 아니라, 프론트가 좁혀준 context를 바탕으로 답변을 보강하는 방식이 가장 안전합니다.
